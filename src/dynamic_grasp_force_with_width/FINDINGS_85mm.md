## 1. ⚠️ REVIEW 의 "±6 g" 는 in-sample 입니다

REVIEW 는 3점을 적합한 뒤 **그 3점에 대한 RMSE** 를 정확도로 내걸었습니다.
그건 "직선이 그 점들을 지나간다"는 뜻이지 **"새 무게를 맞힌다"** 는 뜻이 아닙니다.

우리도 똑같이 하면 **RMSE 0.4 g** 가 나옵니다. 하지만 **캘리브레이션에 안 쓴 무게**로
검증하면 이렇습니다.

```
640 g 상자 (저울). 예측 tau 를 재기 전에 선언했습니다 → -9.464

  ①  파지 후  60초 대기 → dither 10회    tau -9.487  →  644.3 g   오차  +4.3 g
  ②  파지 후 300초 대기 → dither 10회    tau -9.580  →  661.9 g   오차 +21.9 g
                                                        ──────────────────
                        같은 상자·같은 파지, 두 측정이 17.6 g 차이
```

**1회 측정의 실력은 ±20 g 입니다.** 파지력 계산(640 g 에서 3%)엔 충분하지만,
±6 g 이 필요한 용도엔 못 씁니다. 정확도가 더 필요하면 **2회 측정 평균**을 쓰세요.

## 2. ⚠️ R² ≥ 0.95 는 합격선으로 너무 헐겁습니다

점이 몇 개 안 되면 R² 는 거의 항상 통과합니다. 실제로 4점에서:

```
R²  0.9906        ← 통과처럼 보임
LOO-CV  ±52.8 g   ← 실제 예측력. 최악의 점은 101 g 빗나감
```

**LOO-CV 와 구간별 기울기**를 보세요. 구간 기울기가 널뛰면
(우리 경우 -207 → -255 → -158 g/Nm) 그건 비선형이 아니라 **점 하나가 오염된 것**입니다.

## 3. 파지 직후 ~10초는 과도구간입니다 — 읽으면 안 됩니다

```
파지 명령 후
   5s   tau = -10.600      ← 오버슈트
  10s   tau =  -9.540      ← 정착
  15s ~ 300s   -9.540 고정
```

**2초 뒤에 읽으면 빈 그리퍼처럼 보이기까지 합니다.** 294 g 상자를 물자마자 읽었더니
`-5.830` 이 나왔는데, 이는 물체 없는 그리퍼(`-5.86`)와 사실상 같은 값입니다.
**15초 이상 기다리세요.**

## 4. ⚠️ 정적 토크는 안 흔들려도 못 믿습니다

640 g 상자를 물고 **300초간 5초 간격으로 60회** 읽었더니 `-8.878` 이 **한 번도 안 변했습니다.**
완벽하게 안정된 값입니다. 그런데 같은 상태에서 dither 평균은 `-9.580` 이었습니다.

```
정적 (60회 연속 동일)   -8.878
dither 평균             -9.580
차이 0.70 Nm  =  133 g
```

**정적 토크는 마찰 밴드(±0.7 Nm) 안 아무 데나 앉고, 거기 앉으면 미동도 안 합니다.**
"안 흔들리니 믿을 만하다"는 완전히 틀린 직관입니다. **dither+평균만 유효합니다.**

## 5. 폭 의존성 — "±10 mm 면 계수 하나로 충분" 도 낙관적입니다

```
물병  30 mm    -214.2 g/Nm
상자  75 mm    -163.6 g/Nm
상자  85 mm    -189.4 g/Nm      ← 폭 10 mm 에 기울기 16% 변화
```

REVIEW 는 "폭 편차가 ±10 mm 면 계수 하나로 충분"이라 했지만, 75→85 mm 에서
기울기가 **16% 움직였습니다.** 750 g 에 적용하면 100 g 이 넘는 오차입니다.

> 게다가 기울기가 폭에 대해 **단조롭지도 않습니다** (214 → 164 → 189).
> 폭 하나로 매끄럽게 보간할 수 있는 관계가 아닐 수 있습니다.
> **상자 폭이 다양하다면 OCR 에 `width_mm` 을 요청하고 폭별로 캘리브레이션하세요.**

## 6. ❓ 미해결 — 490 g 점이 51 g 빗나갔고 원인을 모릅니다

```
직선상 예측   tau = -8.670
실측          tau = -8.401      →  51 g 오차
```

확인된 산포(±20 g)의 **2.5배**입니다. 과도현상(10초)으로도 설명이 안 됩니다 —
충분히 기다린 뒤 쟀습니다. **원인 모르는 실패 모드가 하나 남아 있습니다.**

**대응: 캘리브레이션할 때 잔차를 항상 확인하세요.** 한 점만 크게 튀면 그 점을 버리고
다시 재세요. 이번엔 그 점을 빼니 4점 RMSE 21.9 g → 3점 RMSE 0.4 g 로 떨어졌습니다.

# 운영 함정 (코드/환경)

## A. 툴 등록은 드라이버 재시작 때 날아갑니다

REVIEW 1-4 에 이미 있지만 **또 당했습니다.** 세션 시작 시 확인하세요.
`info=''` 면 등록이 안 된 겁니다.

```bash
ros2 service call /dsr01/dsr_controller2/tool/get_current_tool \
  dsr_msgs2/srv/GetCurrentTool
```

툴이 없으면 모터 토크가 약 2.5 Nm 달라져 **캘리브레이션이 통째로 무효**입니다.

## B. 🔥 스크립트에서 DSR_ROBOT2 를 쓰면 무한정 매달립니다

`DSR_ROBOT2` 는 import 시점에 서비스 클라이언트를 만들고, 호출할 때
**`wait_for_service` 를 하지 않습니다.** 게다가 `spin_until_future_complete` 에
**타임아웃도 없습니다.**

```python
# DSR_ROBOT2.py:647
def get_robot_mode():
    future = _ros2_get_robot_mode.call_async(req)
    rclpy.spin_until_future_complete(g_node, future)   # ← 타임아웃 없음
```

노드를 만들고 몇 ms 만에 호출하면 **ROS 디스커버리가 아직 안 끝나서** 요청이
허공으로 날아가고 **영원히 매달립니다.** 노트북은 셀 사이 시간 덕에 우연히 피해갑니다.

**게다가 매달린 프로세스가 컨트롤러를 물어서 `ros2 service call` 까지 먹통이 됩니다.**
(로봇이 죽은 줄 알고 한참 헤맸습니다. 프로세스를 죽이니 바로 정상화됐습니다.)

**해결 — 노드 생성 후 필수 서비스를 명시적으로 기다리세요:**

```python
node = rclpy.create_node("...", namespace=ROBOT_ID)
DR_init.__dsr__node = node
from DSR_ROBOT2 import ...          # import 시 클라이언트가 생성됨

NEEDED = ("get_robot_mode", "get_current_posj", "move_joint", ...)
need = [c for c in node.clients if any(k in c.srv_name for k in NEEDED)]
t0 = time.time()
while time.time() - t0 < 20.0:
    pending = [c for c in need if not c.service_is_ready()]
    if not pending:
        break
    rclpy.spin_once(node, timeout_sec=0.1)
if pending:
    raise RuntimeError("디스커버리 실패: " + ", ".join(c.srv_name for c in pending))
```

> ⚠️ **`node.clients` 전체를 기다리면 안 됩니다.** 이 드라이버에 존재하지 않는 서비스가
> 섞여 있습니다 (예: `motion/set_singularity_handling_force`). 쓰는 것만 기다리세요.

## C. 그리퍼 로그는 '반전된' 값을 찍습니다

`gripper_service.py:199-202` 가 하드웨어로 보내기 직전에 stroke 를 뒤집습니다
(실제 하드웨어는 `0=열림 / 750=닫힘` 으로 문서 규약과 반대). 그런데 **로그는 반전 후 값**을
찍습니다.

```
gripper_cmd(750)  (= 규약상 '열기')  →  로그: "gripper cmd: pos=0"
```

**버그가 아닙니다.** 로그만 헷갈리게 돼 있습니다. 명령이 뒤집힌 줄 알고 시간을 썼습니다.

## D. 물체 존재 확인 — 가벼운 물체엔 원리상 안 통합니다

```
빈 그리퍼 (stroke 564 / current 350, 물체 없음)   tau = -5.86 Nm
빈 상자   (52 g)                                  tau = -6.36 Nm
                                                  차이 0.5 Nm
```

무거운 물체는 토크로 유무를 확실히 가릴 수 있지만, **가벼운 물체는 빈 그리퍼와 구분이
안 됩니다.** 자동 판정에만 의존하지 마세요.

# 다음 세션 — 파지력 데이터

무게 저울은 **±20 g 로 검증됐습니다.** 이제 `current = A·질량 + B` 를 구할 차례입니다.

## 확정된 사실

- **`current 350` 이 750 g 을 버팁니다** (85 mm 상자, 60 mm 리프트, 슬립 없음)
- `CURRENT_MAX = 500` 은 여전히 **미실측**입니다 (상자가 찌그러지기 시작하는 값)

## 절차

```
① 툴 등록 확인 (드라이버 재시작했으면 재등록)
② 픽 자세로 이동
③ 상자마다 current 를 올려가며 "버티는 최소값" 찾기
④ planner.fit(trials) → current = A·질량 + B
```
