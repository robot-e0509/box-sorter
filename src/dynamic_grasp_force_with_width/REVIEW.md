# 1. 무게 측정 — 확정된 방식

## 1-1. `actual_motor_torque` 만 실측값입니다

**E0509 에는 관절토크 센서도 F/T 센서도 없습니다.**
`raw_joint_torque` 가 사실 `gravity_torque` 와 **완전히 동일한 모델 계산값**이라,
`external_joint_torque = raw − gravity = 0` 이 **구조적으로 항상 성립**합니다.

| 경로 | |
|---|---|
| `get_tool_force()` | ❌ 위 필드를 읽으므로 **영원히 0** |
| `get_workpiece_weight()` | ❌ 측정 함수가 아님 (`set_` 으로 선언한 값을 돌려주는 게터) |
| `raw_force_torque` | ❌ F/T 센서 미장착 |
| **`actual_motor_torque`** (`read_data_rt`) | ✅ **유일한 실측값** |

## 1-2. 방법

```
질량 = a · tau(J2) + b        ← 절대 토크. 기준선을 따로 빼지 않습니다
```

조건이 전부 같으면 **기준선은 절편 `b` 에 흡수**됩니다.

```
① 기준 물체(빈 페트병 등)를 물고 dither+평균  →  캘리브레이션 점
② 무게를 바꿔가며 3~4점                       →  회귀 (R² 확인)
③ 이후 아무 물체나 물고 measure()             →  질량
```

**500 ml 물병에 물을 채워가며 재는 게 가장 좋습니다.** 폭이 안 바뀌므로
stroke·자세·짜는 힘이 전부 고정되고 **오직 무게만** 변합니다.

## 1-3. ⚠️ 함정 3개 — 여기서 이틀을 잃었습니다

### ① 짜는 반력 — 빈 그리퍼를 기준선으로 쓰면 안 됩니다

빈 그리퍼는 stroke 에 **도달**해서 아무것도 안 누릅니다.
물체를 물면 `current` 로 계속 **누릅니다.** 그 반력이 팔에 모멘트를 만듭니다.
두 상태를 빼면 반력이 남아 무게 신호를 덮어버립니다.

실제로 **부호가 뒤집힌 값**(무게를 달았는데 토크가 줄어듦)이 나왔습니다.

**→ 기준 물체를 쓰세요.** 같은 폭·같은 힘으로 문 상태끼리 비교해야 합니다.

### ② 마찰 히스테리시스 — 팔을 건드릴 때마다 값이 달라집니다

감속비가 커서 마찰이 **약 3 Nm**. 정지 상태의 평형 토크는 **유일하지 않습니다.**

```
가만히 두면 (2분)          변동폭 0.137 Nm  = 양자화 한 칸      ← 안정
같은 물체를 놨다 다시 물면   변동폭 0.8~1.5 Nm ≈ 150~280 g       ← 못 씀
```

**dither 를 그냥 넣으면 오히려 악화됩니다** (0.133 → 0.596 Nm).
하지만 흩어짐이 **치우침 없이 무작위**라서 **평균**을 내면 됩니다.

```
dither(J2 ±1°, 항상 같은 방향에서 복귀) → 매번 새 평형점에 정착 → 독립 표본
5회 평균 → 표준오차 약 10 g
```

### ③ J2 를 쓰세요. J3 는 못 씁니다

```
J2   R² 0.9973   ← 이걸 쓰세요
J3   R² 0.8489   ← 드리프트 심함 (520 g 측정 중 -21.1 → -29.1 로 계속 흘러감)
```

동료분의 **B-4(다중 관절)** 지적대로, **어느 관절을 쓸지는 데이터로 정해야** 합니다.
자세가 바뀌면 최적 관절도 바뀝니다.

## 1-4. 사전 조건 — 툴 등록

**툴 등록은 모터 토크를 약 2.5 Nm(≈780 g) 바꿉니다.**
그런데 **드라이버를 재시작하면 사라집니다.** 즉 **툴 상태가 캘리브레이션의 일부**입니다.

대조 실험으로 확정했습니다.

```
모드 전환만 (manual→autonomous, 툴 그대로)   Δ토크 0.000 Nm   ← 무영향
툴 등록                                        ΔJ3   2.468 Nm   ← 이게 원인
```

**manual 모드에서만** 등록됩니다. `WeightSensor` 가 조건을 검사해서 다르면 **거부**합니다.

## 1-5. ⚠️ 캘리브레이션이 유효한 조건

```
자세      [-13.181, -23.339, 94.612, 15.686, 55.968, -15.458]
그리퍼    stroke 186 명령 · current 350
툴        rh_p12_rn (0.5 kg, cog [0,0,60])
물체 폭   30 mm
```

**하나라도 바뀌면 재캘리브레이션해야 합니다.**
특히 파지 `current` 를 바꾸면 짜는 반력이 달라져 계수가 무효가 됩니다.

## 1-6. 겪은 실패 (다음 사람이 또 겪지 않도록)

- **젖은 물병은 `current 350` 으로도 미끄러집니다.** 두 번 겪었고, 두 번 다 데이터가
  **조용히 오염**됐습니다. (J2/J3 부호가 서로 반대로 나오는 게 신호였습니다)
  → **물기를 반드시 닦으세요.**
- **첫날 아이폰(174 g) 측정이 잘 나온 건 운이었습니다.** 짧은 시간에 연속으로 재서
  마찰 평형이 안 바뀐 것뿐입니다. 몇 분만 지나도 무너집니다.
- 옛 REVIEW 에 썼던 **"stroke 별 빈 그리퍼 기준선 테이블"은 폐기**했습니다 (함정 ①).

# 2. 원본 코드의 버그 — 실행을 막습니다

| 버그 | 내용 |
|---|---|
| **`is_force_closure()`** | 법선이 `[-dzdx, -dzdy, 1]` 이라 z가 항상 +. antipodal(`\|n1+n2\| < 0.2`)이 **수학적으로 불가능** → `select_best_grasp()` 항상 `None` → **`run_task()` 100% 실패**. 직육면체 상자엔 과설계이니 **빼세요** |
| **접촉점** | `box[0], box[2]` 는 **대각선 꼭짓점**입니다 (주석은 "마주보는 변의 중점"). 80×40 사각형에서 89.4 px(대각선)가 나옵니다. `(box[0]+box[1])/2`, `(box[2]+box[3])/2` 가 맞습니다 |
| **힘의 단위** | `default_force = 20.0` (뉴턴)이 `gripper_close(current=20)` 으로 들어갑니다. 권장 `current` 는 200~400 이라 **20은 상자를 무조건 놓치는 값**입니다. 코드 전체에서 `force` 는 그냥 `current` 로 쓰입니다 — **"N" 이름표만 잘못 붙었습니다** |
| **`stability_score()`** | `contact_points`(mm)와 `center_of_mass`(픽셀)를 뺍니다. 값이 무의미합니다 |

> `grasp_v2` 는 뉴턴을 아예 안 씁니다. `current = A·질량 + B` 로 직접 회귀합니다.
> 물리식은 "필요한 힘이 무게에 **비례**한다"는 **형태**만 주고, 계수는 실험이 줍니다.

# 3. 리뷰 반영 내역

| 지적 | 조치 |
|---|---|
| **B-5** 기준선이 그리퍼 위치에 의존 | ✅ 맞습니다. 다만 **더 근본적인 문제(짜는 반력)** 가 있었습니다 → 기준 물체 방식으로 해결 (1-3 ①) |
| **B-1** 다중점 회귀 | ✅ 3~4점 회귀 + R² + 잔차표 + LOO-CV |
| **B-3** `CURRENT_PROBE` 적정성 | ✅ **실제로 문제였습니다.** `100` 은 515 g 을 못 버팁니다 → `350` 사용. 단, 바꾸면 재캘리브레이션 필요 |
| **B-4** 다중 관절 | ✅ **결정적이었습니다.** J3 는 못 쓰고 J2 만 됩니다 (1-3 ③) |
| A-1 서비스 결과 미검증 | ✅ `_call()` 이 `success=False` / timeout 이면 예외 |
| A-2 노이즈 경고가 경고로만 | ✅ `measure()` 가 `(mass, ok)` 반환 → `ok=False` 면 fixed 로 자동 폴백 |
| A-3 조용한 clamp | ✅ MIN 미만은 정보, **MAX 초과는 위험 경고** + `status="over_max"` |
| A-4 `PICK_POS` 휘발 | ✅ `pick_pose.json` 저장/로드 |
| A-5 모드 전환 미확인 | ✅ `set_mode()` 가 폴링해서 확인, 안 되면 예외 |
| A-6 타임스탬프 없음 | ✅ 모든 기록에 `t` |
| A-7 held-out 없음 | ✅ 3~4점이라 held-out 은 무리 → **LOO-CV** |
| C-1 이동 중 미끄러짐 | ✅ 들어올린 뒤 재측정. 질량이 절반 아래면 "놓쳤습니다" |

## ⚠️ D-1 (점진적 그립) 의 하드웨어 제약

방향은 맞다고 봅니다. 다만 두 가지 제약이 있습니다.

- **그리퍼 현재 위치를 읽을 수 없습니다.** 슬립 신호로 position 을 쓰려면

  이동 중 실시간 감지엔 너무 느립니다. **트레이드오프를 실측으로 정해야** 합니다.

## 절차

**1. 띄우기**

```bash
./scripts/robot_up.sh
source install/setup.bash && jupyter notebook
```

`grasp_v2.ipynb` 셀 1~4 (연결 → 모드 헬퍼 → **툴 등록** → 그리퍼 상수).

**2. 픽 자세 티칭** (셀 6~7) — `pick_pose.json` 에 저장됩니다.

> ⚠️ **세션 내내 고정.** 바뀌면 캘리브레이션을 전부 다시 해야 합니다.

**3. 무게 캘리브레이션** (셀 8~9) — 물병에 물 채워가며 3~4점 → `scale.fit()`

> **R² ≥ 0.95** 가 나와야 합니다. 안 나오면 미끄러진 것입니다.

**4. 파지력 데이터** — 상자마다 `current` 를 올려가며

```python
try_current(150, width_mm=30)
reject(0)                       # 미끄러졌으면
try_current(200, width_mm=30)
accept(1)                       # 버텼으면
```

**"버틴 것 중 가장 낮은 `current`" 를 accept 하세요.**

**5. 회귀** — `planner.fit(trials)` → `current = A·질량 + B`

**여기까지 나오면 2단계 완성입니다.**

## 오늘 성공 기준

1. 저울 값과 로봇 측정값이 맞는다 (**R² ≥ 0.95**)
2. `current = A·질량 + B` 가 나온다 (3점 이상)
3. 무게가 다른 상자를 실제로 집어 옮긴다
